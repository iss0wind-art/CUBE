# CUBE — 3D 공간형 코딩 환경 (Architect OS)

> tmux가 화면을 2D 타일로 쪼갠다면, CUBE는 사용자를 큐브 방 안에 넣고
> 여섯 면 전체를 작업 공간으로 쓴다. ORCA(Hundred Rabbits)처럼
> 격자 하나하나가 살아있는 유닛이다.

## 핵심 개념

1. **구체 프랙탈** — 우리는 큐브 안에 있다. 큐브 안에 큐브가 중첩될 수 있다.
2. **ORCA 격자** — 격자 하나하나가 터미널이다.
3. **정면 벽 = 메인 창** — 현재 작업 중인 터미널 (CLI든 GUI든).
4. **좌/우/천장 격자** — 각 셀은 터미널이 될 수도, 메뉴가 될 수도 있다. 사용자 자유.
5. **세계수 (바닥)** — 처음엔 조그만 애기나무 한 그루. 프로젝트가 진행될수록 자란다.
   각 프로젝트의 *진짜* git worktree이며, 브랜치가 3차원으로 뻗어 한눈에 보인다.
   - 아래로 숨겼다 뺐다 자유
   - 2D로 축소해 좌/우측 벽의 큐브 하나에 도킹 가능
   - 좌측 큐브(백엔드 작업)가 세계수의 한 가지와 연결되고,
     우측 큐브(프론트엔드 작업)가 다른 가지를 점유하는 식
6. **큐브 = 서랍** — 언제든 끼웠다 뺐다, 위치 이동 자유.
7. **모델링 바닥** — 3D 모델링 사용자는 바닥을 간단한 모델링 툴로 쓸 수 있다.
8. **마인드맵 연결** — 좌측 1번 큐브를 우측 9번 큐브로 드래그하면 가상의 선이
   나타나고 둘이 연결된다 (프로젝트 연계). 선 표시/숨김은 토글.
9. **후방 포털** — remote-ssh, 다른 차원(호스트)으로의 이동.
   협업자의 연결선들이 포털을 통해 오간다.
10. **별 = 다른 사람의 큐브** — 큐브 밖 우주에 보이는 별들은 전부
    다른 사용자의 큐브다. 별 하나 = 접속 가능한 차원 하나.
    포털 워프 = 그 별(큐브)로의 이동. 멀티플레이어의 세계관.
11. **모바일도 같은 개념.**

## 카메라 UX 결정 (2026-07-16)

- 작업 중에도 회전은 기본 허용 (공간감이 이 툴의 정체성).
- 대신 **잠금 토글** (헤더 자물쇠 버튼 / Ctrl+Shift+L): 회전·줌 동결 + 정면 스냅.
- **모니터 프레임/주변 벽 클릭 = 정면 정렬 스냅** (터미널 내부 클릭은 제외).

## 아키텍처

```
[브라우저]                          [로컬 서버 :3001]
React 19 + CSS 3D (큐브 룸)   ⇄    Express + ws
xterm.js (벽면 터미널)        WS    @lydell/node-pty (PowerShell)
                                    (Phase 2+: git 데이터 수집)
```

- CSS 3D를 유지하는 이유: xterm.js는 DOM이므로 CSS 3D 벽에 그대로 붙는다.
  WebGL 씬이었다면 터미널 임베드가 지옥이 된다.
- 실행: `npm run dev` (web + pty 서버 동시 기동)
- 조작: 휠 = 줌, 휠버튼 드래그 = 시점 회전, 셀 클릭 = 터미널 생성/포커스,
  셀 우클릭 = 터미널 종료

## 카메라 설계 (Phase 1.5)

- **COORDS (기본, 내부 조종석)**: 눈이 포털 벽 바로 앞(로컬 z:+220)에 위치,
  회전 피벗 = 눈. 고개 돌리기 rx[-30,25] / ry[±35] 클램프, 휠 = 뒤로 물러나기만.
- **ORBIT (전지적)**: 기존 외부 궤도 카메라. 자유 회전 + 줌 0..400.
- **Chrome 제약 (중요)**: 시점 평면(z=perspective)을 가로지르는 요소는 통째로
  컬링된다. 그래서 방 정중앙 360° 카메라는 CSS로는 불가 — 옆벽이 눈 평면을
  가로지르는 순간 사라진다. 완전한 내부 360°가 필요해지면 벽을 셀 단위
  개별 평면으로 재구성(three.js CSS3DRenderer가 이 방식)해야 한다.
- 메인 터미널은 벽 전체가 아니라 벽 중앙의 640×560 "모니터 패널" —
  조종석 시점에서 전부 시야에 들어오는 크기.
